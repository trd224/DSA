Very High Frequency Two Pointer Array O(n) time · O(1) space
problem statement
Given a 1-indexed array of integers numbers that is already sorted in non-decreasing order, find two numbers that add up to a specific target.

Return the indices of the two numbers as an integer array [index1, index2] (1-indexed), where 1 ≤ index1 < index2 ≤ numbers.length.

The tests are generated such that there is exactly one solution. You may not use the same element twice.
⚡
Key constraint: The array is already sorted. This is what makes Two Pointer work in O(1) space — no HashMap needed. Use the sorted order to guide your pointers. Returns 0-indexed result.

approach comparison
❌ HashMap (Two Sum I style)
Works but wastes the sorted property. Uses O(n) extra space for no benefit when the array is already ordered.
O(n) timeO(n) space
✅ Two Pointer — Optimal
Left pointer starts at index 0, right at end. Sum too big? Move right left. Sum too small? Move left right.
O(n) timeO(1) space

your solution
javascript 
function twoSum(numbers, target) {
    let left = 0;
    let right = numbers.length - 1;

    while (left < right) {
        let sum = numbers[left] + numbers[right];

        if (sum === target) {
            return [left, right];  // return indices (0-indexed)

        } else if (sum > target) {
            right--;               // sum too big  → move right left

        } else {
            left++;                // sum too small → move left right
        }
    }

    return -1; // no solution found (defensive)
}

console.log(twoSum([2, 7, 11, 15], 9));  // [0, 1]
console.log(twoSum([2, 3, 4], 6));        // [0, 2]
console.log(twoSum([-1, 0], -1));         // [0, 1]

why two pointer works here
🧠
The Sorted Array Guarantee
Because the array is sorted, when sum > target we know for certain that right pointer value is too large — we can safely discard it. When sum < target, left value is too small — safely advance it. This is impossible with an unsorted array.
💡
Why O(1) Space vs HashMap's O(n)
Two Sum I needs a HashMap because we don't know which pairs to check. Two Sum II's sorted order gives us a deterministic decision rule at each step — no storage needed.
✅
Your Code Uses 0-Indexed Output
Your code returns [left, right] — standard 0-indexed JS style. If a problem requires 1-indexed output (like LeetCode 167), just change to [left + 1, right + 1].

interview variants — be ready
Variant 1: Three Sum — Fix one element, run Two Pointer on the rest. Sort first, then for each nums[i] run Two Pointer on [i+1, end].
Variant 2: Four Sum — Two nested loops + Two Pointer inside. O(n³).
Variant 3: Count pairs with sum = target — Same pattern but count all valid pairs, skip duplicates by checking while (L < R && nums[L] === nums[L-1]) L++.
Variant 4: Closest sum to target — Track Math.abs(sum - target) as minimum distance, update result when closer found.
